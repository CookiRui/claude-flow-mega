## 1. 核心理念

### 1.1 为什么需要项目配置

AI 编程助手的效果高度依赖上下文质量。没有结构化的项目文档，AI 每次对话都从零开始理解你的项目，输出质量不稳定。

**目标**：让 AI 像一个刚入职但读完了所有内部文档的新员工——知道项目的原则、规范、架构和常用模式。

### 1.2 三个核心原则

| 原则           | 说明                                                                                     |
|----------------|------------------------------------------------------------------------------------------|
| **渐进式披露** | AI 不需要一次性读完所有文档。先加载原则（宪法），按需加载实现细节（Skills）              |
| **单一信息源** | 每个知识点只在一个地方定义。宪法说"AI 会做错什么"，Rules 补充具体约束，Skills 说"怎么用" |
| **Context 即成本** | 每多加载一行文档就多消耗一份 token。冗余 = 浪费钱 + 挤占有效信息的空间              |

### 1.3 Token 预算意识

| 文件类型     | 加载时机                             | 预算建议                                |
|--------------|--------------------------------------|-----------------------------------------|
| CLAUDE.md    | **每次对话自动加载**                 | < 100 行                                |
| Constitution | **每次对话自动加载**（通过 @import） | 单个 60-150 行，所有子系统合计 < 400 行 |
| Rules        | **每次对话自动加载**                 | 总量 < 200 行                           |
| Skills       | **按需加载**（触发时才读）           | 单个 < 100 行                           |
| Commands     | **用户主动调用**                     | 不限                                    |

**每次对话的"固定成本" = CLAUDE.md + Constitution + Rules**，这部分必须精打细算。

## 2. 四层信息架构

```
┌──────────────────────────────────────────────────────┐
│  Constitution（宪法）                                  │
│  项目特有的核心约束，AI 不知道就会写错的规则              │
│  加载：每次对话 | 修改频率：极低                         │
├──────────────────────────────────────────────────────┤
│  Rules（规则）                                         │
│  宪法的补充细则，宪法推导不出来的具体约束                  │
│  加载：每次对话 | 修改频率：低                            │
├──────────────────────────────────────────────────────┤
│  Skills（技能）                                        │
│  具体框架/API 的使用模板和参考                           │
│  加载：按需触发 | 修改频率：中                            │
├──────────────────────────────────────────────────────┤
│  Commands（命令）                                      │
│  用户可调用的多步骤工作流                                │
│  加载：用户主动调用 | 修改频率：低                        │
└──────────────────────────────────────────────────────┘
```

| 维度       | Constitution        | Rules                    | Skills                | Commands                 |
|------------|---------------------|--------------------------|-----------------------|--------------------------|
| 回答       | "AI 会做错什么"     | "补充什么细节"           | "具体怎么用这个API"   | "执行这个完整流程"       |
| 举例       | "并发用 Actor 模型" | "safego.Go 参数注意事项" | "UniTask 的 API 用法" | "/feature-plan 需求分析" |
| 每次加载   | ✅ 是               | ✅ 是                    | ❌ 按需               | ❌ 手动                  |
| 冗余容忍度 | 零                  | 零                       | 低                    | 可接受                   |

## 3. CLAUDE.md 分层设计

项目中应有多层 CLAUDE.md，形成信息树：

```
project/
├── CLAUDE.md                    ← 根入口：目录结构 + 工作规范 + @import 子系统
├── {subsystem-a}/
│   └── CLAUDE.md                ← 子系统：技术栈 + 核心目录 + @import constitution/rules/skills
├── {subsystem-b}/
│   └── CLAUDE.md                ← 子系统：同上
├── {subsystem-b}/{module}/
│   └── CLAUDE.md                ← 模块级：职责 + 目录 + 状态机 + 扩展规范
└── {shared-layer}/
    └── CLAUDE.md                ← 共享层：协议格式 / 公共库 / 配置规则
```

### 各层内容与行数预算

| 层级             | 预算    | 内容                                                  |
|------------------|---------|-------------------------------------------------------|
| 根 CLAUDE.md     | < 50 行 | 目录结构 + 工作规范 + @import 子系统和根 constitution |
| 子系统 CLAUDE.md | < 80 行 | 技术栈 + 核心目录 + @import constitution/rules/skills |
| 模块级 CLAUDE.md | 不限    | 模块职责 + 架构图 + 状态机 + 扩展指南（按需加载）     |

### @import 语法

```markdown
@{subsystem-a}/CLAUDE.md
@.claude/rules
@.claude/constitution.md
@.claude/skills/*/SKILL.md
```

**避免循环引用**：单向树形结构，每个文件只被 import 一次。

## 4. Constitution 宪法层

### 4.1 核心原则

宪法**只定义 AI 不知道的、项目特有的、反直觉的核心约束**。

**写入标准**：如果删掉某条，AI 的默认行为是否会产生错误代码？
- 是 → 写入宪法
- 否 → 不需要

### 4.2 高价值 vs 低价值条目

**该写的**（AI 不知道就会写错）：

| 条目         | 为什么 AI 会做错                                       |
|--------------|--------------------------------------------------------|
| 核心架构模式 | AI 不知道你用 Manager/ECS/MVVM，会按自己的理解组织代码 |
| 模块通信约束 | AI 会直接 import 其他模块的类，破坏解耦                |
| 并发模型     | AI 可能用共享内存+锁，而你要求 Actor 消息传递          |
| 技术选型强制 | AI 会用语言默认方案（Debug.Log）而非项目封装（MLog）   |
| 热路径性能   | AI 的"正常写法"在 Update 中产生 GC                     |
| 数据访问约束 | AI 会直接写 SQL，而非通过 Repository 层                |

**不该写的**（AI 默认就会遵守）：

| 条目           | 原因                       |
|----------------|----------------------------|
| 简单性/YAGNI   | Claude Code 系统指令已内置 |
| 不要空 catch   | 通用编程常识               |
| 注释写"为什么" | 通用编程常识               |
| 命名规范       | 放 Rules 层，不是宪法级    |

### 4.3 写作要点

1. **"不可协商"标记** — 用于绝对规则
2. **代码示例成对** — ✅ 正确 和 ❌ 错误 并排，比纯文字有效 10 倍
3. **前向引用 Skill** — `详见 {skill-name} skill`，不在宪法里写细节
4. **4-7 条** — 太多 AI 记不住（使用 §1、§2 编号格式）
5. **HTML 注释引导** — `<!-- 填写指南 -->` 帮助填写

### 4.4 强制执行协议

宪法治理条款中包含 5 项强制执行协议：

1. **Skill 强制加载** — 任务匹配 Skill 触发条件时，必须加载并遵循
2. **子代理约束继承** — subagent 必须先读 constitution + 相关 Skills
3. **确认门不可跳过** — 即使用户催促也不得跳过确认环节
4. **完成前验证** — 声明完成前必须过 `verification` skill
5. **违规即修** — 发现违宪代码立即标记并修复

这些协议通过宪法治理条款获得最高优先级，确保 AI 在长上下文或复杂任务中不会偷懒跳步。

## 5. Rules 规则层

### 5.1 黄金法则：Rules ≠ Constitution 的复制品

```
❌ 宪法说"禁止裸 go 关键字"，Rules 又写一遍 → 浪费 token
✅ 宪法说"禁止裸 go 关键字"，Rules 补充"safego.Go 的参数传递注意事项" → 有价值
```

**判断标准**：如果删掉某条 Rule，AI 是否仍能从宪法中推导出来？能 → 删掉。

### 5.2 写作要点

1. **回溯宪法** — 开头标注"per Constitution §X"
2. **承认例外** — `允许的例外` 比绝对禁止更实用
3. **以检查清单结尾** — 方便 AI 自检

### 5.3 预算控制

| 指标                 | 建议值   |
|----------------------|----------|
| 单个 Rule 文件       | < 80 行  |
| 所有 Rule 文件总行数 | < 200 行 |
| Rule 文件数量        | 1-3 个   |

超过 200 行 → 要么和宪法重复了，要么该下沉到 Skills。

> 模板：`../template/.claude/rules/coding-style.md`

## 6. Skills 技能层

### 6.1 设计理念

Skills 是**按需加载的参考文档**，只在 AI 判断需要时才加载，不占每次对话的固定预算。

目录结构：
```
.claude/skills/{skill-name}/
├── SKILL.md              # 入口（触发描述 + 核心规则，< 100 行）
└── references/detail.md  # 详细参考（完整 API + 代码模板，不限）
```

### 6.2 触发描述的写法（最关键）

触发描述决定了 AI 什么时候加载这个 Skill，**写得好坏直接影响命中率**。

```yaml
# ❌ 模糊 → AI 不触发
description: "关于网络的规范"

# ✅ 精准（关键词 + 场景 + 触发条件）→ AI 准确触发
description: "网络通信规范：HTTP请求、Socket连接、Protobuf消息收发、断线重连。当涉及网络请求、消息收发、断线重连、网络错误处理时使用。"
```

**公式**：`"{主题}：{关键词1}、{关键词2}、{关键词3}。当涉及{场景1}、{场景2}、{场景3}时使用。"`

### 6.3 数量建议

| 阶段 | 数量  | 覆盖                        |
|------|-------|-----------------------------|
| 初期 | 5-10  | 核心架构                    |
| 中期 | 15-25 | + 业务流程                  |
| 成熟 | 20-40 | 不宜超过 50（太多 AI 选错） |

> 模板：`../template/.claude/skills/_template/`

### 6.4 内置方法论 Skills

模板内置两个**通用方法论 Skill**：

**`tdd`** — 测试驱动开发
- 强制 RED-GREEN-REFACTOR 循环，不可跳步
- 每个循环覆盖一个行为点，≤ 5 分钟
- 按项目类型适配（Unity PlayMode / Jest / Pytest 等）

**`verification`** — 完成前验证
- 功能完整性 → 测试验证 → 宪法合规 → 代码质量 → 影响范围
- 5 项检查全部 PASS 才能声明"完成"
- 输出结构化验证摘要表

这两个 Skill 不是建议，而是通过宪法治理条款**强制执行**的。

## 7. Commands 命令层

### 7.1 最有价值的两个 Command

经过实际验证，投入产出比最高的：

**`/feature-plan-creator`** — 需求分析 → 技术方案 → 微任务拆解
- 流程：读文档 → 多轮确认需求 → 输出 plan.md → 拆解为 ≤5 分钟微任务
- 每个微任务指定目标文件 + 完成标准 + TDD 标记
- 价值：避免 AI 在不理解需求的情况下直接写代码，且粒度可控

**`/bug-fix`** — Bug 诊断 → 回归测试 → 修复 → 验证 → 经验固化
- 流程：定位 → 根因分析 → 等用户确认 → **先写回归测试** → 修复 → verification 验证 → 更新文档/Skill
- 价值：理解根因后再动手，TDD 方式修复，修完后固化经验

### 7.2 设计要点

1. **确认门（Gate）** — 分析和执行之间必须 `AskUserQuestion`，这是最重要的设计
2. **禁止事项** — 明确写"不要做什么"，比"要做什么"更有约束力
3. **合宪性自检** — 执行后对照宪法检查
4. **完成前验证** — 声明"完成"前必须走 `verification` skill
5. **子代理约束继承** — 派发 subagent 时，子代理必须先读 constitution + 相关 Skills
6. **微任务粒度** — 技术方案必须拆解为 ≤5 分钟的可验证微任务

## 8. 内容归属决策树

写了一条新规范，不确定放哪层？

```
这条规范 AI 不知道就会写错吗？
├── 是 → Constitution（宪法）
│         它回答的是"为什么"还是"怎么做"？
│         ├── 为什么 → 留在宪法
│         └── 怎么做 → 移到 Skills，宪法只留原则 + 前向引用
│
└── 否 → 它是每次对话都需要知道的吗？
          ├── 是 → Rules
          │         宪法能推导出这条吗？
          │         ├── 能 → 不需要，删掉
          │         └── 不能 → 保留，标注"基于宪法第X条"
          │
          └── 否 → 它是关于特定框架/API 的使用方法吗？
                    ├── 是 → Skills
                    └── 否 → 多步骤工作流？
                              ├── 是 → Commands
                              └── 否 → 对应模块的 CLAUDE.md
```

### 常见错误归属

| 内容                               | ❌ 错误放置  | ✅ 正确放置      |
|------------------------------------|--------------|------------------|
| "并发用 Actor 模型"                | Rules        | Constitution     |
| "UniTask.RunOnThreadPool 参数说明" | Constitution | Skills           |
| "MLog.Info?.Log 调用格式"          | Constitution | Skills           |
| "Bug 修复完整流程"                 | Rules        | Commands         |
| "某模块的状态机定义"               | 根 CLAUDE.md | 模块级 CLAUDE.md |

## 9. 其他 Claude Code 能力速查

以下能力不属于四层架构，但对项目配置有帮助。

| 能力              | 用途                                                   | 配置位置                                        | 要点                                                   |
|-------------------|--------------------------------------------------------|-------------------------------------------------|--------------------------------------------------------|
| **Memory**        | 跨会话记忆（当前开发阶段、踩坑记录、个人偏好）         | `~/.claude/projects/{project}/memory/MEMORY.md` | 自动加载，< 200 行                                     |

## 10. 反模式与踩坑总结

### 10.1 Constitution 和 Rules 重复（最常见）

**症状**：宪法写了"禁止裸 go 关键字"，Rules 又写一遍，每次对话浪费 token。

**解决**：Rules 只写宪法未覆盖的补充细节。定期审计，删除能从宪法推导出的 Rules。

### 10.2 宪法里写通用常识

**症状**：宪法里有"YAGNI"、"不要空 catch"、"注释写为什么"，每次对话加载但不改变 AI 行为。

**解决**：只保留项目特有的、AI 默认行为会违反的约束。通用编程常识全部删除。

### 10.3 @import 循环引用

**症状**：同一内容被加载两次。

**解决**：画 @import 依赖图，确保单向树形。根 CLAUDE.md @import 子系统 CLAUDE.md，子系统负责 @import 自己的 constitution。

### 10.4 Skill 描述太模糊导致不触发

**症状**：写了 Skill 但 AI 从来不用。

**解决**：description 中堆砌触发关键词和场景（见 §6.2）。

### 10.5 Skill 太多导致选错

**症状**：50+ 个 Skill，AI 经常加载错误的。

**解决**：合并窄范围 Skill，保持 40 个以内。

### 10.6 Command 缺少确认门

**症状**：AI 在 bug-fix 流程中直接改代码，没等用户确认。

**解决**：分析和执行之间必须插入 `AskUserQuestion`。

### 10.7 AI 在复杂任务中偷懒跳步

**症状**：长上下文或多步任务中，AI 跳过测试、跳过验证、"觉得差不多了"就收工。子代理不遵守项目约束。

**解决**：
1. 宪法治理条款中加入强制执行协议（5 项不可协商条款）
2. 使用 `tdd` skill 强制 RED-GREEN-REFACTOR 循环
3. 使用 `verification` skill 强制完成前验证
4. 子代理约束已集中在宪法治理条款中，Commands 不再重复声明

### 10.8 任务粒度太粗导致失控

**症状**：AI 接到"实现用户系统"这样的大任务，一口气写大量代码，质量不可控，出错难以定位。

**解决**：`/feature-plan-creator` 强制将方案拆解为 ≤5 分钟微任务，每个任务指定目标文件和完成标准。

## 11. 新项目启动

### 自动初始化（推荐）

```bash
# 1. 复制框架到项目
cp -r template/ your-project/
cp -r scripts/ your-project/scripts/

# 2. 在项目目录下启动 Claude Code，运行：
/init-project
```

`/init-project` 会自动扫描代码库，生成所有配置文件（constitution、rules、.claudeignore、Skills 等），只需确认一次分析结果。

### 手动初始化

如果 `/init-project` 的结果不满意，或偏好手动控制：

- [ ] 复制 `template/` 到项目根目录
- [ ] 替换 `CLAUDE.md`、`.claude/constitution.md`、`.claude/rules/coding-style.md` 中的 `{占位符}`
- [ ] 自定义 `.claudeignore`

### 持续优化

- [ ] AI 做错了什么就补什么（Skill / Rule / Constitution）
- [ ] 每月审计一次（Rules 冗余、Skill 失效）
- [ ] Bug 修复后评估是否需要更新 Rule 或 Skill

## 12. 持续优化

### 反馈闭环

```
AI 输出不符合预期？
├── 不知道项目框架怎么用 → 创建 Skill
├── 违反了项目规则 → 完善 Constitution
├── 细节写法不对 → 补充 Rule
├── 反复犯同一个错 → 写入 Memory
└── 输出符合预期 → 继续开发
```

**关键心态**：不要提前设计所有 Skill。最好的配置来自实战——AI 做错了什么，就补什么。
