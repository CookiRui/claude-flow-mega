## 30 秒速览

```
任务进来
  ↓
判断复杂度 S/M/L/XL
  ↓
S → 直接执行 + 验证 + 提交              （1 分钟搞定）
M → 分解 → 逐个执行 + 检查点 + L1/L2    （5-30 分钟）
L → 审查目标 → DAG 分解 → 并行执行 → 三级验证 → 元学习（30 分钟+）
XL → L + 外部决策介入                   （跨会话）
```

**两个核心理念**：
1. **不杀鸡用牛刀**：80% 的日常任务（S/M 级）走快速路径，把重型流程留给 L/XL 级
2. **不达目的不停止**：单次会话预算耗尽不是终点，持久化循环自动从 WIP 恢复继续推进

## 第一步：复杂度分流

接到任何任务，**第一件事**是判断复杂度级别。

### 判定速查表

| 问自己的问题                  | 是   | 否     |
|-------------------------------|------|--------|
| 只改 1 个文件？               | → S  | 继续 ↓ |
| 改 2-5 个文件，验证条件明确？ | → M  | 继续 ↓ |
| 跨模块？涉及架构决策？        | → L  | 继续 ↓ |
| 需要产品/战略层面判断？       | → XL | 回到 L |

### 实际例子

| 任务                                 | 级别 | 推荐模型 | 理由                                |
|--------------------------------------|------|----------|-------------------------------------|
| "把 PlayerManager 的 hp 改成 health" | S    | Haiku    | 单文件重命名                        |
| "给登录接口加验证码"                 | M    | Sonnet   | 改 2-3 个文件，有明确验证条件       |
| "游戏帧率优化到稳定 60fps"           | L    | Opus     | 跨模块，需要 profiler 分析 + 多方案 |
| "要不要从 Unity 迁移到 Unreal"       | XL   | Opus     | 战略决策，影响全项目                |

## S 级任务：快速路径

**流程**：执行 → L1 验证 → git commit。不走审查、不走分解、不走元学习。

```
1. 直接用 Edit/Write 工具修改代码
2. 自检：改动是否符合预期
3. git commit -m "checkpoint: {简要描述}"
```

如果执行过程中发现复杂度比预估高 → **立即升级到 M 或 L**。

---

## M 级任务：标准路径

**流程**：简化分解 → 逐个执行 + 检查点 → L1/L2 验证。

```
1. 快速分解为 2-5 个子任务（不需要 DAG，线性即可）
2. 为每个子任务定义完成条件
3. 逐个执行：
   a. 执行子任务
   b. git commit（检查点）
   c. L1 验证（自检）
4. 全部完成后 L2 验证（关键路径才需要）
```

## L/XL 级任务：完整路径

走完整 8 层流程。这是框架的核心价值所在。

### Step -1: 生成代码地图 + 选模型

```bash
python scripts/repo-map.py /path/to/project
```

**选模型**：L/XL 级主 Agent 用 **Opus**，子 Agent 按类型选：
- 编辑类子任务 → `Agent(model="haiku")`
- 搜索/分析子任务 → `Agent(model="sonnet")`
- 对抗验证 L2 → `Agent(model="sonnet")`

### Step 0: 目标审查

```
检查清单：
  □ 物理可行性：技术上能做到吗？
  □ 清晰度：验证条件是否无歧义？
  □ 隐含 tradeoff：达成目标要牺牲什么？用户知道吗？
  □ 范围合理性：目标与预算匹配吗？
```

### Step 1: DAG 分解 + 预检

```
用 TaskCreate 创建每个子任务：
  - 名称
  - 完成条件（必须机器可判定）
  - 依赖哪些任务
  - 预估复杂度
```

**分解完成后立即预检**：覆盖性？独立性？可验证性？粒度合理？依赖正确？

### Step 2: 并行执行 + 冲突检测

```
找出所有依赖已满足的任务
  ↓
冲突检测：会改同一个文件/模块吗？
  ├─ 无冲突 → Agent() 并行执行
  ├─ 有冲突但可隔离 → Agent(isolation="worktree")
  └─ 有冲突不可隔离 → 退化为串行执行
```

### Step 3: 每个子任务的执行循环

```
while 未通过验证 and 尝试次数 < 3 and 预算充足:
    1. 查长期记忆 → 有历史经验吗？
    2. 制定策略 → 注入残值和约束
    3. 信心 < 0.3 → 先搜索学习
    4. 信心仍 < 0.3 → 升级给人类（Level 2）
    5. 执行
    6. git commit（检查点）
    7. 回归验证 → 之前通过的还通过吗？
    8. 多级验证 → L1/L2/L3
    9. 通过 → 下一个任务
       失败 → 萃取残值 → 换策略重试
```

### Step 4: 动态重规划

每完成一批子任务后，检查剩余任务是否仍然有效。结果推翻了假设？→ 重新分解。

### Step 5: 元学习

任务完成后，分析执行轨迹，存入 `memory/meta_*.md`。

## 安全网：检查点与回滚

每个子任务完成后：
```bash
git commit -m "checkpoint: {task_name} — {简要结果}"
```

回归验证失败时：
```bash
git revert {commit_hash}
```

## 验证体系：三级验证

| 场景            | 验证级别 | 操作                               |
|-----------------|----------|------------------------------------|
| 常规子任务      | L1       | 自检：改动符合预期 + 测试通过      |
| 关键路径/高风险 | L2       | L1 + 启动子 Agent 用边界条件挑战   |
| 最终交付        | L3       | L1 + L2 + 从原始目标出发跑完整场景 |

### L2 对抗验证示例

```
提示词模板：
"你是一个严格的代码审查者。以下是刚完成的修改：{diff}。
请检查：
1. 边界条件是否处理了？
2. 有哪些隐含假设可能出错？
3. 与其他模块的集成会有风险吗？
只关注真正的问题，不要吹毛求疵。"
```

## 长对话保护：防止上下文丢失

### 策略 1: 关键信息写到文件

每完成 3-5 个子任务，把进度写到 WIP 文件。

### 策略 2: 重型搜索用子 Agent

大量搜索/分析交给子 Agent 处理，只返回精简结果。

### 策略 3: 预算过半时强制外化

当预算消耗超过 50%，自动把当前完整状态写到 WIP 文件。

## 元学习：越用越聪明

### 元策略文件格式

```markdown
---
name: meta_{problem-domain}
description: Meta-strategy for {domain} tasks — optimal decomposition/search/verification
type: feedback
---

## {Problem Domain} Tasks

### Best Strategies
- {insight-1} (success rate: X%)
- {insight-2}

### Verification Level
- Recommend L{N}, because {reason}

### Budget Allocation
- {per-phase percentages}

### Last Validated
- Date: YYYY-MM-DD
- Project: {project-name}
- Result: {specific-result}
```

### 何时更新元策略

| 触发条件             | 操作                           |
|----------------------|--------------------------------|
| L/XL 任务完成后      | 分析轨迹，更新或创建 meta 文件 |
| 验证日期超过 30 天   | 标记待验证，下次使用时验证     |
| 同一域的策略矛盾     | 保留最近验证通过的版本         |
| 发现历史策略不再适用 | 更新策略 + 标注失效原因        |

## Hooks 配置：Lint/Test 反馈闭环

不只是"系统跑个脚本"（单向），而是 **编辑 → lint/test → 失败 → 自动反馈给 AI → AI 修复 → 再跑**（双向闭环，借鉴 Aider）。

### 推荐配置

在项目的 `.claude/settings.json` 中：

```jsonc
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "command": "bash scripts/lint-feedback.sh"
      },
      {
        "matcher": "Write",
        "command": "bash scripts/lint-feedback.sh"
      }
    ]
  }
}
```

`scripts/lint-feedback.sh` 自动检测项目类型（Node/Python/C#/Go/Rust），对修改的文件跑 lint，失败时将结构化错误输出到 stderr，Claude Code 读取后自动修复。

### 闭环对比

| 场景       | 旧方案（单向 Hooks）       | 新方案（双向闭环）                         |
|------------|----------------------------|--------------------------------------------|
| lint 失败  | 用户看错误，手动让 AI 修   | 错误自动反馈 AI → AI 自动修 → 循环直到通过 |
| test 失败  | AI 记得跑测试（不可靠）    | 编辑后自动跑相关测试 → 失败自动反馈        |
| 危险命令   | AI 自觉避免（不可靠）      | 执行前脚本拦截（100% 可靠）                |

## 常见场景速查

### 场景 1: 修一个 bug

```
1. 判定级别：通常 M 级
2. 定位 bug → 确认根因
3. 先写回归测试（TDD）
4. git commit（检查点）
5. 修复 bug
6. 跑测试 → 通过
7. git commit
8. L1 验证（测试通过 + 功能正常）
```

### 场景 2: 新增一个功能模块

```
1. 判定级别：L 级
2. 目标审查：功能边界清晰吗？与现有模块冲突吗？
3. DAG 分解：数据层 → 逻辑层 → 接口层 → 测试 → 文档
4. 分解预检：覆盖性？独立性？粒度？
5. 逐个执行 + 检查点
6. 关键路径 L2 验证
7. 最终 L3 验证
8. 元学习：这类功能的最佳开发顺序是什么？
```

### 场景 3: 性能优化

```
1. 判定级别：L 级
2. 目标审查："稳定 60fps"→ 哪个场景？平均还是最低？
3. DAG 分解：Profiler → 方案设计 → 实施（可并行）→ 验证
4. 查元策略：memory/meta_performance_optimization.md
5. 执行 + 检查点
6. L2 验证：不同场景都达标？
7. L3 验证：连续 5 分钟跑 benchmark
8. 元学习：更新性能优化元策略
```

### 场景 4: 任务做到一半要中断

```
1. 保存 WIP：执行轨迹 + 残值 + 剩余任务 + 建议下一步
2. 下次从 WIP 恢复继续
```

### 场景 5: AI 信心不足

```
信心 > 0.8  → AI 全自动执行，你只看结果
信心 0.5-0.8 → AI 通知你进展，继续执行
信心 0.3-0.5 → AI 给出 2-3 个选项，你选方向
信心 < 0.3  → AI 完整交接：进度 + 尝试记录 + 建议
```

## 踩坑记录与对策

### 踩坑 1: 所有任务都走完整流程

**现象**：改个变量名也走 8 层，浪费时间。
**对策**：第一步永远是判定复杂度。S 级直接执行。

### 踩坑 2: 分解粒度不对

**现象**：子任务太粗 → 执行中才发现有歧义；太细 → 管理开销超过执行收益。
**对策**：探索型任务粗分，边做边细化；实施型任务细分，每个 ≤5 分钟。

### 踩坑 3: 忘了做回归验证

**现象**：修了 B 破了 A，到最后才发现。
**对策**：配置 Hooks 自动跑测试。至少每个子任务后手动跑一次相关测试。

### 踩坑 4: 长对话丢失早期信息

**现象**：做了 10 个子任务后，忘了前面的关键发现和约束。
**对策**：每 3-5 个子任务写一次检查点摘要。预算过半时强制外化到 WIP 文件。

### 踩坑 5: 方案失败后从头来过

**现象**：方案 A 失败，直接抛弃所有中间结果重新开始。
**对策**：萃取残值。失败中有排除的假设、可复用的代码、收窄的问题范围。

### 踩坑 6: 元策略只存在脑子里

**现象**：上次总结的经验，下次又忘了。
**对策**：执行完 L/XL 任务后，立即更新 `memory/meta_*.md`。有文件才能跨会话复用。
